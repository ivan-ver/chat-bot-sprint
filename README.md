**Вывод для LLM**: Для данного кейса локальная модель предпочтительна из-за требований к конфиденциальности данных. 
Это единственный способ гарантировать, что спецификации проектов и детали интеграций не будут обработаны в стороннем облаке. 
Использование облачных API возможно только для неконфиденциальных данных, что усложнит архитектуру.

**Вывод для векторной БД:** ChromaDB — явный победитель для данного кейса. Он предоставляет готовое решение, что 
ускоряет разработку и снижает затраты на поддержку. FAISS больше подходит для случаев, когда нужна максимальная 
производительность на железе и когда он уже встроен в существующую платформу.

### Рекомендуемая конфигурация сервера
Для обработки ~25k документов и ожидаемого количества запросов от ~100-200 сотрудников:

* **CPU**: 8-16 ядер (e.g., Intel Xeon Silver или аналогичный AMD EPYC). Важно для быстрого инференса эмбеддингов и работы модели.
* **RAM**: 32-64 GB. LLM ~7B параметров в 4-битной квантованной версии требует ~4-6 GB. Остальная память — для загрузки эмбеддинг-модели, самой ОС, векторной БД и кэша.
* **GPU (Сильно рекомендуется)**: NVIDIA RTX 4090 (24GB) или A10 (24GB). Это ускорит инференс и эмбеддингов, и LLM в разы (в 10-20 раз по сравнению с CPU), что критично для отклика системы в реальном времени. 24GB памяти хватит для загрузки 7B-модели и обработки запросов.
* **Storage**: Быстрый SSD NVMe объёмом 500GB-1TB (для ОС, моделей, индексов и логов).
* **Сеть**: Стабильный канал внутри AWS VPC для доступа к внутренним API (Confluence, GitHub).

Без GPU система будет работать, но время ответа может достигать 10-30 секунд, что негативно скажется на опыте пользователей.


### Итоговые рекомендации для QuantumForge Software
#### Варианты конфигурации

* **Вариант 1:** Полностью локальное развёртывание (Recommended)
LLM: Локальная, квантованная модель (e.g., Mistral 7B или Llama 3 8B). Качество сопоставимо с GPT-3.5 для RAG-задач.

    * Эмбеддинги: Локальная модель all-MiniLM-L6-v2 или sentence-t5-large.
    * Векторная БД: ChromaDB, развёрнутая в том же контейнере/на том же сервере.
    * Инфраструктура: Выделенный инстанс в AWS EC2 (например, g5.2xlarge с GPU T4 16GB или лучше g6.xlarge с A10) в рамках существующего VPC. Это обеспечит безопасность, низкую задержку до других корпоративных сервисов и легкое управление.

| Преимущества                                       | Недостатки                                                                                                         |
|----------------------------------------------------|--------------------------------------------------------------------------------------------------------------------|
| 1) Максимальная безопасность и контроль.           | 1) Требует навыков DevOps для настройки и поддержки.                                                               |
| 2) Предсказуемая стоимость (hourly rate инстанса). | 2) Качество ответов может быть slightly ниже, чем у топовых облачных моделей (но достаточно для внутренних задач). |
| 3) Данные не покидают периметр AWS VPC компании.   |                                                                                                                    |


* **Вариант 2**: Гибридное развёртывание
    * Эмбеддинги + Поиск: Локально (как в Варианте 1) для безопасности.
    * LLM: Облачный API (OpenAI GPT-4-Turbo или YandexGPT) только для тех запросов, где не фигурируют конфиденциальные данные. Это требует реализации классификатора на основе полученных из векторной БД чанков для определения конфиденциальности запроса.

| Преимущества                                                                         | Недостатки                                                  |
|--------------------------------------------------------------------------------------|-------------------------------------------------------------|
| 1) Возможность получить высочайшее качество ответов для неконфиденциальных вопросов. | 1) Двойная стоимость (инфраструктура + API-вызовы).         |
|                                                                                      | 2) Высокая архитектурная сложность.                         |
|                                                                                      | 3) Риск ошибки в классификации и утечки данных.             |
|                                                                                      | 4) Не рекомендуется из-за высоких рисков для данного кейса. |


* **Вариант 3**: Полностью облачное (на базе AWS Bedrock)
    * LLM & Эмбеддинги: Использование моделей через AWS Bedrock (e.g., Anthropic Claude, Mistral, Llama).
    * Векторная БД: ChromaDB на EC2 или Pinecone (но он тоже облачный).

| Преимущества                                                                                                       | Недостатки                                                                                                                                                          |
|--------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 1) Полная управляемость, высокий уровень сервиса.                                                                  | 1) Стоимость API-вызовов Bedrock.                                                                                                                                   |
| 2) Данные обрабатываются в том же облачном провайдере (AWS), что и продакшн-среда, что может упростить compliance. | 2) Данные всё равно обрабатываются в managed-сервисе, а не на нашем инстансе. Юридически это может быть приемлемо, но технически контроль меньше, чем в Варианте 1. |



##### Рекомендуется выбрать Вариант 1.

**Обоснование:**
  * Соответствие требованиям безопасности: Это главный аргумент. Компания работает с данными промышленных объектов и интеграциями с SCADA-системами, что по умолчанию подразумевает высокий уровень конфиденциальности. Локальное развёртывание полностью снимает эти риски.
  * Контроль над затратами: Ежемесячная стоимость мощного EC2-инстанса с GPU будет фиксированной и предсказуемой, в отличие от потенциально растущих счетов от OpenAI при активном использовании бота.
  * Производительность: Современные локальные модели (Mistral, Llama 3) при правильной настройке промптов и качественной подготовке чанков показывают отличные результаты для задач точного поиска и ответа по базе знаний.
  * Интеграция в экосистему: Развёртывание в том же AWS VPC, что и основная инфраструктура, обеспечит низкую задержку при доступе к источникам данных (например, для периодического переиндексирования документов из Confluence/GitHub) и простоту управления доступом.
